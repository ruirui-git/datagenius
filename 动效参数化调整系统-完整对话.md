# 用「可视化调参面板」消除设计-开发动效协作摩擦

## 背景：一个反复发生的低效循环

做过动效还原的同学，大概都经历过这样的场景：

设计师在 Figma 或 Principle 里做好了一版卡片悬浮动效，导出录屏发给开发。开发拿到视频，开始"对齐"——`transition duration` 是多少？`easing` 是什么曲线？hover 的时候卡片上移了几像素？阴影怎么变化的？

这些问题，很多时候设计工具要么没有直接暴露，要么和代码实现的参数体系不一样，开发只能凭肉眼判断，凭感觉去试数值。

还原完了，设计师来验收——"这个感觉不太对，弹性不够，我想要更丝滑一点"。于是开发改了一版发截图/视频，设计师再看……这个循环可能反复跑好几圈。

**核心摩擦在于：设计侧和开发侧对动效参数的"语言"不统一，中间有一道翻译损耗。**

---

## 三个具体痛点

### 痛点 1：设计工具暴露的参数不完整

设计工具（Figma、Principle、After Effects）的动效参数，和前端代码（framer-motion、CSS transition、GSAP）能控制的参数不是一一对应的。

比如一个卡片 hover 效果，代码里可以独立控制：
- hover 时 Y 轴上移量（`yFactor`）
- hover 时的 scale 放大倍数（`scaleFactor`）
- 两层阴影各自的 blur、偏移、透明度（6 个参数）
- 推开效果里相邻卡片、次相邻卡片的位移（2 个参数）

但设计工具里，这些往往是耦合在一起的视觉结果，设计师调的是"感觉"，不是代码层面的具体数值。

### 痛点 2：参数体系不一致，需要转译

即使设计工具能导出参数，也需要开发做一层转译。

Figma 的 `easing` 是预设名称（"Ease Out"），代码里需要的是 `cubic-bezier(0.4, 0, 0.2, 1)`。设计稿里的阴影是静态的颜色值，代码里需要的是随 hover 状态插值变化的动态函数。这种转译本身就是一个误差来源。

### 痛点 3：验收反馈只能靠截图录屏

最低效的环节在这里：开发实现完，设计师验收，如果不符合预期，设计师能做的只有：
- 截图标注 + 文字描述
- 录屏对比原版和实现版
- 面对面 pair review

但这些方式都有信息损耗。"弹性感不够"、"阴影太重了"这样的反馈，映射到代码参数上，开发还是要靠猜和试。

---

## 新思路：把代码参数开放给设计师

既然核心问题是"设计师无法直接操控代码层面的动效参数"，那为什么不直接把参数开放出来？

思路是：**在页面上做一个可视化的参数调节面板，设计师可以通过拉滑杆实时调整动效的每一个代码参数，调到满意后，把参数值直接复制给开发（或者直接提交代码）。**

这样：
- 不需要设计工具和代码之间的参数转译
- 设计师看到的就是代码跑出来的真实效果
- 调完即可用，没有来回沟通成本

## 实现：FanCardEditor 参数面板

### 场景描述

Demo 里有一组 AI Agent 卡片，四张卡片呈扇形排列，hover 时有推开、上浮、阴影渐变等动效。这个动效用到了 24 个代码参数，覆盖从布局到物理效果的各个维度。

![卡片扇形动效示意]

### 参数体系设计

将 24 个参数按语义分成 8 组：

```
扇形布局     overlapX              卡片间距
扇形角度     rotateOuter/Inner     外圈/内圈卡片旋转角
             outerCardOffset       外圈卡片 Y 轴下沉
Hover 推开   dist1X / dist2X       相邻/次相邻卡片推开距离
悬浮物理     hoverHeight           归一化 hover 强度（驱动所有效果）
             yFactor               Y 轴上浮系数
             scaleFactor           放大系数
Hover 投影   shadowBlur1Range      主阴影 blur 变化量
             shadowY1Range         主阴影 Y 轴变化量
             shadowAlpha1Range     主阴影透明度变化量
             （次阴影同上共 3 个）
静态投影     restShadowY/Blur/Alpha 默认状态阴影
信息区毛玻璃 infoFromOpacity/ToOpacity  渐变上下透明度
             infoBlur / infoSaturate    模糊 / 饱和度
动画时间     fanTransitionDuration      扇形过渡时长
             hoverTransitionDuration    hover 过渡时长
```

这里有一个关键的设计决策：**引入归一化的 `hoverHeight`**。

hover 效果里，Y 轴位移、scale、阴影增强是相互关联的——现实中物体被"托起来"，高度越高，影子越淡越扩散，物体越大。如果把这三类效果分别独立暴露，调起来会很难找到协调感。

解决方案是用一个归一化的 `hoverHeight`（0-1）作为统一驱动，其他参数则是各自的"乘数"：

```typescript
function deriveHoverPhysics(h: number, config: FanCardsConfig) {
  const y = -h * config.yFactor;                         // 线性上浮
  const scale = 1 + h * config.scaleFactor;              // 线性放大
  const shadowBlur = restBlur + h * config.shadowBlur1Range;  // 阴影渐增
  const shadowAlpha = restAlpha + h * config.shadowAlpha1Range;
  // ...
}
```

这样设计师只需要调 `hoverHeight` 就能整体控制"hover 强度"，再用各个乘数微调各维度的权重。

### 面板 UI 设计

面板本身是一个可拖动的毛玻璃浮层，有两种状态：
- **展开态**：260px 宽，最高 60vh，8 组参数可折叠，底部有代码预览和复制按钮
- **收起态**：右下角一个 36px 圆球，不遮挡页面

状态切换用 framer-motion 的 AnimatePresence 实现，有明确的形变动画（面板收缩成圆球，圆球展开成面板），视觉上有连续性。

```typescript
// 面板收起退场：缩小到右下角，变成圆形
exit={{
  opacity: 0,
  scale: 0.12,
  y: "68vh",
  x: 85,
  borderRadius: 200,
  backgroundColor: "rgba(0,0,0,0.82)",
}}
transition={{ duration: 0.55, ease: [0.4, 0, 0.65, 0.1] }}

// 圆球退场：放大飞向面板位置
exit={{
  opacity: 0,
  scale: 2.5,
  y: "-65vh",
  backgroundColor: "rgba(255,255,255,0.72)",
}}
transition={{ duration: 0.45, ease: [0.2, 0.8, 0.3, 1] }}
```

每个参数条目包括：中文标签 + 代码键名 + 当前值 + 重置按钮（值非默认时出现）+ 进度条样式的 range input。

还有一个小细节：面板 hover 时，会自动把其中一张 Agent 卡片设为 hover 状态——这样设计师在调 hover 相关参数时，不需要同时悬停卡片，可以专注操作滑杆。

### 代码导出

调完参数，点「复制代码」，剪贴板里就是可以直接用的 TypeScript 常量：

```typescript
export const DEFAULT_FAN_CONFIG: FanCardsConfig = {
  overlapX: -20,
  rotateOuter: 11,
  rotateInner: 4,
  outerCardOffset: 38,
  dist1X: 40,
  dist2X: 24,
  hoverHeight: 0.30,
  yFactor: 40,
  scaleFactor: 0.30,
  // ...共 24 个参数
};
```

粘贴进代码，动效立刻更新。零误差，零转译，零来回沟通。

## 这个方案解决了什么

回到最初的三个痛点：

**痛点 1（参数不完整）**：设计师直接操作代码参数，没有中间层，参数是完整的。

**痛点 2（需要转译）**：不需要翻译，调的就是代码里的值。

**痛点 3（验收靠截图录屏）**：设计师自己调到满意，直接输出参数值，开发零成本同步。

更深的变化是**协作模型的改变**：从"开发实现 → 设计验收 → 反馈 → 开发修改"的串行循环，变成了"设计直接在生产代码上调参 → 输出参数 → 开发一键同步"。设计师变成了动效参数的直接作者，而不是远程指挥者。

---

## 适用场景与边界

这种方式特别适合：
- 组件级别的精细动效（hover、过渡、微交互）
- 参数维度多、调感觉成本高的效果（如本例的 24 个参数）
- 设计师和开发在同一产品团队，有一定技术协作意愿
- Vibe coding / 快速迭代场景，设计和开发边界模糊

不太适合：
- 纯叙事性动效（Lottie 类，时间轴驱动，参数语义和代码差异大）
- 需要复杂绑定的全页面过渡动效
- 设计师完全不接触代码的传统分工模式

---

## 更大的可能性

这次只是在一个 Demo 里做了验证，但这个思路可以推广：

1. **设计系统级别的参数面板**：把整个设计系统里的 motion token（duration、easing、物理参数）都开放出来，设计师可以全局调整。

2. **Storybook 集成**：把参数面板做成 Storybook addon，所有组件的动效参数都可以在组件文档页直接调。

3. **AI 辅助生成初始参数**：把设计师的语言描述（"更弹一点"、"快速但不突兀"）翻译成参数调整建议，设计师做最终确认。

4. **参数版本管理**：记录每次调参的快照，可以对比、回滚，像 Git 一样管理动效演进历史。

---

## 小结

动效协作的摩擦，本质上是两个角色在使用不同的"语言"描述同一件事。设计师说的是"感觉"，开发写的是"数值"。消除这道摩擦，不一定需要更好的设计工具或更聪明的开发——**有时候只需要把数值的控制权直接交给感觉的拥有者**。

Vibe coding 让这件事变得可行：在 AI 辅助下，一个开发可以在几小时内搭出这样的参数面板，成本远低于反复的设计-开发沟通循环。当工具的建造成本足够低，很多以前"不值得做"的协作基础设施，就变得值得做了。

# 设计师直接调代码参数——一次消除 Design to Code 摩擦的尝试

## 那个反复发生的低效循环

做动效的同学大概都经历过这样的场景：

设计师在 Figma 里做好了一版卡片悬浮效果，录屏发给开发。开发拿到视频开始还原——transition 多少毫秒？easing 是什么曲线？hover 的时候卡片上移了几像素？阴影怎么渐变？

很多参数，设计工具根本不会告诉你。开发只能凭肉眼猜，凭感觉试。

还原完，设计师来验收——"这个感觉不太对，再轻盈一点。"开发改一版，发截图。设计师再看——"好了一点，但弹性还是不够……"

这个循环，可能跑三圈、五圈，直到大家都累了，妥协了，上线了。

**问题不是开发不认真，也不是设计师要求高。问题是：设计师感受到的"感觉"，和开发操作的"代码参数"，之间永远隔着一层翻译。**

---

## 三层摩擦，每一层都在损耗

仔细拆开来，这道翻译分三层：

**第一层：设计工具的参数不够用。** 设计师在工具里调的是视觉结果，但代码里控制一个 hover 效果，可能涉及十几个独立参数——上浮高度、放大比例、两层阴影各自的模糊半径和透明度、相邻元素的推开距离……这些代码层面的参数，设计工具几乎不会直接暴露。设计师无法给出开发真正需要的东西。

**第二层：参数体系不一致，需要转译。** 即使能从设计工具里导出一些数值，也不能直接用。Figma 里的 "Ease Out"，到了代码里是 `cubic-bezier(0.4, 0, 0.2, 1)`。设计稿里静态标注的阴影颜色，到了代码里是随交互状态实时插值的动态函数。翻译本身就是误差来源。

**第三层：验收反馈只能靠描述，开发还是得猜。** 最低效的一环：开发实现完，设计师验收不对，能做的只有——截图标注、文字描述、录屏对比。"再轻一点"、"弹性感不够"、"阴影太重了"这些反馈，映射到具体的代码数值上，开发依然只能靠猜和试。

**三层摩擦叠加，每一次协作都是一次信息损耗。最后上线的动效，往往是一个"差不多能接受"的版本，而不是设计师真正想要的那个感觉。**

---

## 换一种思路：把参数直接给设计师调

如果摩擦的根源是"设计师无法直接操控代码参数"，那最直接的解法就是：**把代码参数以设计师能用的方式开放出来。**

不是导出数值文档，不是开培训教设计师写代码，而是：**做一个可视化的参数面板，挂在真实部署的产品页面上，设计师打开链接，拖动滑杆，实时看到效果，调到满意，参数直接输出为代码。**

这就是我在做 WeData AI Demo 时探索的方式。

---

## 具体怎么做的

Demo 里有一组 Agent 卡片，四张卡片呈扇形排列，hover 时有推开、上浮、阴影渐变等效果。这个动效背后有 24 个代码参数，从卡片间距、旋转角度，到悬浮时的物理感、阴影层次、毛玻璃效果，全部可调。

我做了一个参数编辑面板，把这 24 个参数按语义分成 8 组，每个参数对应一根滑杆，拖动时效果实时更新。

关键的设计决策有两个：

**一、参数要有语义，不能是裸数字。** 比如 hover 效果里，上浮、放大、阴影增强三件事其实是一个整体——物体越"高"，影子越淡越扩散，物体显得越大。所以我用一个"hover 强度"的统一旋钮来整体控制这三者，再用各自的"系数"微调每个维度的权重。设计师不需要理解三者的数学关系，调一个旋钮就能感受整体变化，再精细打磨。

**二、这个模块部署在线上，而不是只在本地跑。** 这是最关键的一点。面板不是开发自己用的调试工具，而是部署到实际可访问的 URL 上。设计师打开链接就能调，不需要配置开发环境，不需要懂代码。调好了，点「复制代码」，剪贴板里就是可以直接用的参数值，开发粘贴进去，效果立即同步，一个字节的误差都没有。

**三、这个模块做到了完全解耦。** 参数面板和核心动效代码之间，只有一个干净的接口：24 个参数值的配置对象。面板不侵入动效的实现逻辑，动效也不感知面板的存在。当设计参数确认之后，删除面板模块、把确认好的参数值写入代码，两行操作，整个工具就从生产代码里消失了，不留任何痕迹。这让它能够安全地活在真实的产品里，而不是只能作为一次性的原型。

---

## 这个方式解决了什么

**摩擦一（参数不够用）**：设计师调的就是代码里真实存在的参数，没有中间层，参数是完整的。

**但我觉得你实际走的是路径 B，而且路径 B 才是这个能力真正有意思的地方。**

原因是：如果还要先回 Figma 设计动效，那 Figma 和代码之间的翻译损耗又回来了。Figma 里做的动效，本身就可能和代码实现不一致——这正是你一开始说的痛点。

**所以完整流程应该是这样的：**

**第一步：描述动效意图，AI 生成代码**

设计师不需要画出动效，而是用语言描述——"四张卡片扇形排列，hover 时被悬浮的卡片上浮并放大，旁边的卡片被推开，要有投影变化"。AI 生成可运行的代码。

**第二步：看效果，反复调整到"基本对"**

代码跑起来，设计师看真实效果。大方向不对就继续描述让 AI 改——"推开的幅度太大了"、"投影要更柔和"。这一步是粗调，用自然语言和 AI 来回就行，目标是让动效"大致是那个意思"。

**第三步：识别需要精调的参数，生成面板**

到了"大致对但感觉还差一口气"的阶段，自然语言的颗粒度不够了。"再柔和一点"到底是多柔和？说不清。

这时候设计师让 AI 把影响"感觉"的参数暴露出来做成面板。但不是全部暴露——设计师要判断哪些参数真正影响感觉，哪些可以锁定默认值，哪些应该绑定联动。这是我们前面聊的"参数整合"环节。

**第四步：部署到测试环境，精调**

面板和产品一起部署上线。设计师（以及 PM、开发、任何相关人）打开 URL，通过拉杆精调。

这一步的核心是：**设计师不再用语言描述感觉，而是直接用手去找感觉。** 拖一下，看一下，不对就拖回来。连续的、实时的、无损的。直到"就是这个感觉"。

**第五步：导出参数，应用到代码**

点复制，拿到精确的代码常量，替换默认值。动效定稿。

**第六步：移除面板**

删掉编辑器模块，业务代码不受影响。产品上线。
